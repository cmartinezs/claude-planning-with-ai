Configure the smoke-test plan used before human code review in software projects.

Reference workflows:
- `.planning/WORKFLOWS/02-EXECUTION-WORKFLOWS/GENERATE-DOCUMENT.md`

## Arguments

`$ARGUMENTS` — format: `[--blank]`
- `--blank` — create a scaffold and exit so the user can fill it manually.
- No flag — ask guided questions and populate the file from answers; use repository inference to prefill where possible.

## Steps

1. Read `.planning/config.yml` if it exists. Extract `project.type` and `software.smoke_tests_file` (default: `SMOKE-TESTS.md`).
2. Determine the output file path under `.planning/`: `${software.smoke_tests_file}` unless the config says otherwise.
3. Inspect the repository for stack clues before asking questions:
   - build files (`package.json`, `pom.xml`, `build.gradle*`, `pyproject.toml`, `go.mod`)
   - runtime files (`docker-compose.yml`, `compose.yml`, `Procfile`, `.env.example`)
   - migration/schema files (`flyway/`, `db/migrations/`, `prisma/`, `migrations/`)
   - health or smoke endpoints, CLI entrypoints, and test runners
4. If `--blank` was passed, write the template below with placeholders intact, update `software.smoke_tests_file` in `.planning/config.yml` if needed, and stop.
5. In interactive mode, ask the following questions one at a time. Accept "skip" or a blank answer to leave the placeholder in place.
   1. Which stack(s) does this project use?
   2. What supporting services must be started before the app can run locally?
   3. What is the normal local start or build command?
   4. What schema, migration, or database checks must pass before review?
   5. If this project has an ORM or generated DB client, what static consistency command validates database schema/migrations against ORM models/entities/generated client?
   6. What local runtime command starts the environment needed for persistence smoke checks?
   7. What are the smallest smoke checks that prove the changed surface works?
   8. What evidence should be shown to the human reviewer?
6. Prefill the template from repository inference when the answers are incomplete. Prefer concrete commands and file paths over generic notes.
7. Write the populated smoke-test plan to `.planning/${software.smoke_tests_file}`.
8. If `.planning/config.yml` exists and the file path changed, update `software.smoke_tests_file` to match the generated file.
9. Report the file created, which sections were inferred, and which sections still need manual completion.

## Template

```markdown
# Smoke Test Plan

> Generated by `/plan-smoke-config` or created manually.

---

## Stack Summary

- **Primary stack:** [fill in]
- **Supporting services:** [fill in]
- **Start command:** [fill in]
- **Required environment variables:** [fill in]

---

## Execution Order

1. Start the required supporting services.
2. Build or compile the project locally.
3. Start the app, worker, or API with the project-local configuration.
4. Run the minimal smoke checks that prove the changed surface boots and responds.
5. Capture the evidence needed for human review.

---

## Smoke Checks

| # | Check | How to validate |
|---|-------|----------------|
| 1 | Supporting services are ready | [fill in] |
| 2 | Application compiles and starts | [fill in] |
| 3 | Connectivity or schema validation passes | [fill in] |
| 4 | Changed surface responds correctly | [fill in] |
| 5 | No startup or migration regressions are visible | [fill in] |

---

## Database / ORM Consistency

Required for tasks that modify database structure, migrations, schema files, seed/bootstrap data, ORM models/entities, generated clients, repositories tied to ORM models, or persistence configuration.

| # | Check | How to validate |
|---|-------|----------------|
| 1 | Static DB/ORM consistency | [migration/schema validate, ORM generate/check, typecheck/compile, or manual mapping review command] |
| 2 | Local runtime environment | [command to start DB/services and app/worker locally; ask the human if not inferable] |
| 3 | Persistence smoke check | [minimal non-destructive check for migration/bootstrap and changed persistence path] |

---

## Notes For Automation

- Prefer non-destructive commands.
- If this project has migrations, seed data, ORM models/entities, or generated DB clients, include both static consistency checks and local runtime persistence smoke checks here.
- If the project has no backend or no external services, replace the rows with the closest reproducible local evidence.

---

## Human Review Gate

- Human developer review is required after smoke checks pass.
- If corrections are requested, rerun the impacted smoke checks before asking for approval again.
```
