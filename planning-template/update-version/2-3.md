# Update 2.x -> 3.7.0: Bring Older Planning Workspaces to the Current Baseline

## Goal

Bring any planning workspace that is still on the `2.x` line up to the `3.7.0` baseline.

This migration covers the full set of 3.x changes currently recorded in the changelog:

- v3.0.0 foundation changes
- v3.1.0 smoke-test planning support
- v3.2.0 training workspace layout fix
- v3.4.0 retrospective, edge-case, layered git, and update-version workflows
- v3.5.0 monorepo workspace boundaries, child-planning coordination, direct task branch setup, and DB/ORM validation safeguards
- v3.6.0 child-planning worktree isolation and worktree-prefixed branch naming
- post-merge local branch cleanup for layered task/story branches
- v3.7.0 optional PDR creation, deterministic test-suite generation, logging policy, and richer template examples

## Scope

Inspect and update:

- `.planning/README.md`
- `.planning/GUIDE.md`
- `.planning/GLOSSARY.md`
- `.planning/TRACEABILITY-GLOBAL.md`
- `.planning/config.yml`
- `.planning/SMOKE-TESTS.md`
- `.planning/LOGGING.md`
- `.planning/scripts/generate-test-suite.sh`
- `.planning/_template/`
- `.planning/WORKFLOWS/`
- `.planning/TUTORIAL/`
- `.planning/active/`
- `.planning/finished/`
- any repo-level docs that mirror the planning command inventory or template structure

Do not edit:

- `.git/`
- dependency folders
- generated outputs
- source code or product documentation outside the planning system unless a link to a planning artifact must be corrected and the user explicitly confirms it

## Discovery

Run these searches from the project root:

```bash
rg -n "\b(plan-next|plan-doctor|plan-audit-docs|plan-smoke-config|plan-edge-case|plan-retrospective|plan-update-version|plan-decision|plan-test-suite|SMOKE-TESTS|LOGGING|TEST-SUITE|retrospective|STANDBY|story branch|task branch|git.base_branch|scope-NN|plan-scope|plan-scope-skip|doc-scope|pdr-)\b" .planning .releases README.md CHANGELOG.md docs .page planning-template skills
```

If `.releases/` does not exist, omit it from the search.

Classify matches before editing:

- **Planning-system addition**: new command, workflow, template, or doc that should exist in 3.7.0.
- **Versioned metadata**: version badges, package versions, changelog links, and command counts.
- **Legacy drift**: stale `scope` terminology or old commands that should be converted to `story`.

## Apply In Order

When updating a `2.x` workspace to `3.7.0`, apply the 3.x changes in this order:

1. v3.0.0 foundation
2. v3.1.0 smoke-test configuration
3. v3.2.0 layout and training polish
4. v3.4.0 retrospective and update-version workflows
5. v3.5.0 workspace-boundary, monorepo coordination, layered task git, and DB/ORM validation safeguards
6. v3.6.0 child-planning worktree isolation and branch-prefix safeguards
7. v3.7.0 PDR, logging, and test-suite safeguards

If a later step already exists in the target workspace, leave it intact and only apply missing pieces.

### v3.0.0 foundation

Ensure the workspace has:

- `plan-next`, `plan-doctor`, and `plan-audit-docs` command references where applicable
- the project-guidance template folder under `planning-template/WORKFLOWS/06-PROJECT-GUIDANCE/`
- `planning-template/config.yml` defaults for `terminology`, `autonomy.level`, and `integrations.external_issues`
- risk and external issue fields in planning templates
- docs and tutorials that reference the current story-based model rather than old scope terminology

### v3.1.0 smoke-test configuration

Ensure the workspace has:

- `/plan-smoke-config`
- `.planning/SMOKE-TESTS.md`
- story and task instructions that reference the smoke-test plan before completion

### v3.2.0 layout polish

Ensure the workspace has:

- the updated training workspace layout guidance
- any related UI or tutorial copy that references the revised split-pane behavior

### v3.4.0 retrospective and update-version workflows

Ensure the workspace has:

- `/plan-edge-case`
- `/plan-retrospective`
- `RETROSPECTIVE-RAW.md`
- `RECORD-EDGE-CASE`
- `/plan-update-version`
- the `planning-template/update-version/` catalog
- the current command counts and version badges

### v3.5.0 workspace and execution safeguards

Ensure the workspace has:

- planning command guidance that uses only the current directory's `./.planning/` and never searches parent directories for `.planning/`
- monorepo parent/child coordination guidance: parent plannings coordinate, child artifact plannings own child implementation, and parent `01-expansion.md` records linked child plannings
- `01-expansion.md` template section `## Linked Child Plannings`
- `/plan-task` behavior that creates or reuses the story integration branch when invoked directly, before creating the task branch
- `/plan-story` behavior as optional orchestration for atomized stories, not a required pre-step and not a fallback atomizer
- post-merge local cleanup behavior: after each task PR is merged into the story branch, delete the local task branch with `git branch -d <task-branch>` from an updated story branch checkout
- post-merge local cleanup behavior: after the story PR is merged into `git.base_branch`, delete the local story branch with `git branch -d <story-branch>` from an updated base branch checkout
- guidance that remote branch deletion remains a PR/repository policy action, and merged branch cleanup must not use forced deletes
- task templates, smoke-test templates, and validation commands that require `Database / ORM Consistency Check` for DB/ORM changes
- `/plan-atomize` behavior that adds an explicit DB/ORM validation task after DB/ORM implementation tasks
- docs/tutorials that describe task PRs merging incrementally into the story branch before the final story PR targets `git.base_branch`

### v3.6.0 child planning worktree safeguards

Ensure the workspace has:

- guidance that git-enabled child plannings coordinated by a parent run in dedicated sibling worktrees
- examples that use generic placeholders only, such as `../<worktree-prefix>` and `<worktree-prefix>/story-NN-<slug>`
- `/plan-expand` behavior that creates or reuses child planning worktrees before creating linked child planning files
- `/plan-task`, `/plan-story`, and `/plan-done` behavior that preserves the worktree prefix before the story/task branch portion
- `CHECK-STORY-CONTEXT` behavior that recognizes prefixed child worktree branches and strips the prefix before locating `story-NN-*.md`

### v3.7.0 PDR, logging, and test-suite safeguards

Ensure the workspace has:

- `/plan-decision` guidance for optional Project Decision Records, with PDR creation only when a decision is explicit, accepted, cross-cutting, and supported by enough evidence
- `PDR-TEMPLATE.md` at the planning root and no default blank `pdr-*.md` artifact copied into every new planning
- concrete example values and rows in planning, expansion, story, task, traceability, retrospective, PDR, smoke-test, and idea-capture templates
- `/plan-test-suite`
- `.planning/scripts/generate-test-suite.sh`
- planning, story, and task `TEST-SUITE` artifacts or references to generated `test-suites/<task>-test-suite.md` files where applicable
- `/plan-task`, `/plan-atomize`, task templates, and validators that treat generated test-suite gates as first-class execution evidence
- `.planning/LOGGING.md`
- `/plan-task`, `/plan-atomize`, task templates, and validators that require intelligent logging evidence for software code tasks, including stack-specific recommendation and human decision when no logging mechanism is defined
- `/plan-task` git behavior that commits, pushes, opens or reuses the task PR, waits for external review, pushes requested corrections to the same PR, and marks the task `DONE` only after review approval
