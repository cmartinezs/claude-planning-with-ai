## Goal

Bring any planning workspace that is still on the `2.x` line up to the `3.10.7` baseline.

This migration covers the full set of 3.x changes currently recorded in the changelog:

- v3.0.0 foundation changes
- v3.1.0 smoke-test planning support
- v3.2.0 training workspace layout fix
- v3.4.0 retrospective, edge-case, layered git, and update-version workflows
- v3.5.0 monorepo workspace boundaries, child-planning coordination, direct task branch setup, and DB/ORM validation safeguards
- v3.6.0 child-planning worktree isolation and worktree-prefixed branch naming

- v3.7.0 optional PDR creation, deterministic test-suite generation, logging policy, and richer template examples
- v3.8.0 deterministic validation and documentation script delegation
- v3.9.0 deterministic release-management script delegation
- v3.10.0 deterministic reporting, atomization, story utility, task, lifecycle, initialization, update-version, and `/plan-story` helper delegation

## Discovery

Run these searches from the project root:

```bash
rg -n "\b(plan-next|plan-doctor|plan-audit-docs|plan-smoke-config|plan-edge-case|plan-retrospective|plan-update-version|plan-decision|plan-test-suite|plan-atomize|doc-generate|planning-check|planning-report|planning-atomize|planning-story|planning-task|planning-archive|planning-done\.mjs|planning-init\.mjs|release\.mjs|update-version\.mjs|SMOKE-TESTS|LOGGING|TEST-SUITE|retrospective|STANDBY|story branch|task branch|git.base_branch|scope-NN|plan-scope|plan-scope-skip|doc-scope|pdr-)\b" .planning .releases README.md CHANGELOG.md docs .page planning-template skills
```

If `.releases/` does not exist, omit it from the search.

Classify matches before editing:

- **Planning-system addition**: new command, workflow, template, or doc that should exist in 3.10.7.
- **Versioned metadata**: version badges, package versions, changelog links, and command counts.
- **Legacy drift**: stale `scope` terminology or old commands that should be converted to `story`.

### v3.5.0 workspace and execution safeguards

Ensure the workspace has:

- planning command guidance that uses only the current directory's `./.planning/` and never searches parent directories for `.planning/`
- monorepo parent/child coordination guidance: parent plannings coordinate, child artifact plannings own child implementation, and parent `01-expansion.md` records linked child plannings

- `/plan-task` behavior that creates or reuses the story integration branch when invoked directly, before creating the task branch

### v3.10.1 dashed Markdown table column access fix

Ensure the workspace has the current template copy of:

- `.planning/scripts/planning-check.mjs`
- `.planning/scripts/planning-report.mjs`

These scripts must read dashed Markdown table columns with bracket notation, including `row['depends-on']`, `row['external-issue']`, and `row['external-id']`. Do not keep dotted access such as `row.depends-on`, because JavaScript parses it as subtraction.

### v3.10.2 Story Summary numeric ID extraction fix

Ensure the workspace has the current template copy of:

- `.planning/scripts/planning-check.mjs`
- `.planning/scripts/planning-report.mjs`

These scripts must derive story IDs from the documented Story Summary table's numeric `#` column, such as `01` -> `story-01`, before falling back to scanning legacy row text. Do not require the `Story` slug column to contain a literal `story-NN` token.

### v3.10.3 Story Summary area rendering fix

Ensure the workspace has the current template copy of:

- `.planning/scripts/planning-report.mjs`

Planning reports must derive Story Detail area values from the documented Story Summary `SDLC Phase(s)` column, `Area` or `Repository Area` aliases, and the story file's `## Area` fallback before rendering `-`.

### v3.10.4 Story Summary insertion boundary and DB/ORM false-positive fixes

Ensure the workspace has the current template copy of:

- `.planning/scripts/planning-story.mjs`
- `.planning/scripts/planning-check.mjs`

`planning-story.mjs planning-add-story` must insert new story rows only inside the `## Story Summary` section, even when later tables such as `External Issue Mapping` contain numeric or story-like rows. `planning-check.mjs` must not treat explicit no-database/no-ORM task notes as DB/ORM changes, and generic words such as `repository`, `table`, or `column` should only trigger DB/ORM validation when they appear near DB/ORM context.

### v3.10.5 git-compatible task branch naming

Ensure the workspace has the current template copy of:

- `.planning/scripts/planning-task.mjs`
- `.planning/scripts/planning-story.mjs`
- `.planning/scripts/planning-done.mjs`
- `.planning/GUIDE.md`
- `.planning/_template/01-expansion.md`
- `.planning/WORKFLOWS/04-SUB-WORKFLOWS/CHECK-PLANNING-CONTEXT.md`

Task branches must use sibling refs named like `<story-branch>--task-NN-<slug>`, not nested refs like `<story-branch>/task-NN-<slug>`, because Git cannot keep both `refs/heads/<story-branch>` and `refs/heads/<story-branch>/task-NN-<slug>` at the same time. Task PRs still target the story branch.

### v3.10.6 Summary Evidence requirement for review-only tasks

Ensure the workspace has the current template copy of:

- `.planning/_template/02-deepening/task-NN-name.md`
- `.planning/scripts/planning-atomize.mjs`

Review-only tasks must include `## Summary Evidence` with reviewed scope, evidence artifact, conclusion, and links or snippets as needed. Longer prose artifacts should be Markdown (`.md`), and code evidence must use fenced snippets with language labels.

### v3.10.7 Frontend Design Plan requirement for frontend/UI implementation tasks

Ensure the workspace has the current template copy of:

- `.planning/_template/02-deepening/task-NN-name.md`
- `.planning/WORKFLOWS/02-EXECUTION-WORKFLOWS/ATOMIZE-STORY.md`

The plan must cover idea-to-code flow, view behavior, UI/UX principles, wireframe or equivalent representation, functional mockup before real activity, component pattern, page logic, business logic, external communication/services/APIs/libs, and reuse/modify/create decisions for every affected view, component, service, or library.
