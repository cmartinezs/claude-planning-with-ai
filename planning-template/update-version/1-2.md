# Update 1.x -> 2.x: Scope Terminology to Story Terminology

## Goal

Bring any `.planning/` workspace on the `1.x` line up to the current `2.x` planning baseline.

This migration is cumulative for the `1 -> 2` major transition. Patch and minor releases on the `2.x` line must update this file instead of creating exact-version migration files.

The initial `2.0.0` baseline migrated planning-system artifacts from the old `scope` model to the `story` model. This file updates commands, file and folder names, workflow references, headings, links, and explanatory text where `scope` means the old planning execution unit.

## Version Mapping

- Source line: any `1.x` workspace, including `1.4.0`.
- Target line: the current `2.x` baseline recorded in this file.
- Filename contract: `1-2.md`.
- `/plan-update-version 1.4.0 2.0.0` and other exact labels must resolve to this major-pair migration.
- Only a new adjacent target major should create a new migration file; for example, the `3.x` transition uses `2-3.md`. Do not create skip migrations such as `1-3.md`.

## Scope

Inspect and update:

- `.planning/README.md`
- `.planning/GUIDE.md`
- `.planning/GLOSSARY.md`
- `.planning/TRACEABILITY-GLOBAL.md`
- `.planning/_template/`
- `.planning/WORKFLOWS/`
- `.planning/TUTORIAL/`
- `.planning/active/`
- `.planning/finished/`
- root `.planning/NNN-*` folders still in INITIAL state
- `.releases/` only if release files explicitly reference planning artifacts by old `scope` names

Do not edit:

- `.git/`
- dependency folders
- generated outputs
- source code or product documentation outside `.planning/`, unless a link to a planning artifact must be corrected and the user explicitly confirms it

## Discovery

Run these searches from the project root:

```bash
rg -n "\b(plan-scope|plan-scope-skip|doc-scope|scope-NN|scope-[0-9][0-9]?|EXECUTE-SCOPE|ATOMIZE-SCOPE|NEXT-SCOPE|Scope Summary|Scopes|Scope|scopes|scope)\b" .planning .releases
```

If `.releases/` does not exist, search `.planning/` only.

Classify matches before editing:

- **Planning unit**: old execution unit, should become `story`.
- **Command/workflow name**: old plugin command or workflow, should become the v2 name.
- **External technical meaning**: OAuth, permission, API, CSS, JavaScript, or business-domain `scope`; do not replace automatically.

## File and Folder Renames

Rename planning unit files and folders under every planning:

| Old | New |
|-----|-----|
| `02-deepening/scope-NN-name.md` | `02-deepening/story-NN-name.md` |
| `02-deepening/scope-NN-name/` | `02-deepening/story-NN-name/` |

Preserve the numeric suffix and slug exactly.

After each rename, update Markdown links and path references that pointed to the old path.

## Text Replacements

Apply these replacements only when the term refers to the planning system:

| Old | New |
|-----|-----|
| `/plan-scope` | `/plan-story` |
| `/plan-scope-skip` | `/plan-story-skip` |
| `/doc-scope` | `/doc-story` |
| `plan-scope` | `plan-story` |
| `plan-scope-skip` | `plan-story-skip` |
| `doc-scope` | `doc-story` |
| `EXECUTE-SCOPE` | `EXECUTE-STORY` |
| `ATOMIZE-SCOPE` | `ATOMIZE-STORY` |
| `NEXT-SCOPE` | `NEXT-STORY` |
| `scope-NN` | `story-NN` |
| `scope-01`, `scope-02`, etc. | `story-01`, `story-02`, etc. |
| `Scope Summary` | `Story Summary` |
| `Scope` table column | `Story` table column |
| `scopes` as planning units | `stories` |
| `scope` as a planning unit | `story` |

Use sentence-case and pluralization naturally:

- `Scope` -> `Story`
- `Scopes` -> `Stories`
- `scope` -> `story`
- `scopes` -> `stories`

## Do Not Replace Blindly

Leave a match unchanged when it clearly refers to:

- OAuth scopes
- API permission scopes
- lexical scope or language/runtime scope
- CSS scoping
- database transaction scope
- business-domain scope that is not a planning execution unit

If a line is ambiguous, keep it unchanged and report it as residual follow-up.

## Structural Checks

After edits:

1. Every story file under `02-deepening/` should start with `story-NN-`.
2. Every atomized task folder under `02-deepening/` should start with `story-NN-`.
3. `01-expansion.md` should list `story-NN` IDs, not `scope-NN` IDs.
4. Links in README, expansion, story, traceability, and tutorial files should point to existing files.
5. Commands suggested to users should use `/plan-story`, `/plan-story-skip`, and `/doc-story`.
