## Required Levels

| Level | Use For |
|-------|---------|
| TRACE | Very detailed execution flow, disabled by default outside local debugging |
| DEBUG | Diagnostic state useful during development or incident investigation |
| INFO | Important business or workflow milestones |
| WARN | Recoverable anomalies, retries, fallbacks, degraded behavior |
| ERROR | Failed operations requiring attention, with exception/error context |
| FATAL | Process-level or system-level failure that prevents safe continuation |

If the stack's logging framework does not support `FATAL`, map it to `ERROR` with a clear fatal marker or event field.

---

## Correlation And Traceability

Every code task should preserve or add enough logging to follow execution across calls:

- request/correlation id enters at the boundary and is propagated downstream
- external calls include dependency name, operation, status, latency, and correlation id
- async/event handlers log message id, topic/queue, attempt number, and correlation id
- persistence operations log business-safe identifiers, operation, outcome, and latency when useful
- retries, fallbacks, validation failures, and security-relevant denials are logged at appropriate levels
- errors include exception type and safe context, without leaking sensitive payloads

## Stack Recommendations When No Mechanism Exists

| Stack Signal | Suggested Mechanism |
|--------------|---------------------|
| Java / Spring / Maven / Gradle | SLF4J with Logback or Log4j2, MDC for correlation ids, JSON encoder when logs are machine-consumed |
| Node.js / TypeScript | Pino or Winston, AsyncLocalStorage for correlation ids, structured JSON logs |
| Python | stdlib `logging` plus `contextvars`, or `structlog` for structured logs |
| Go | `log/slog` or Zap with context propagation |
| .NET | Microsoft.Extensions.Logging with Serilog for structured logs |
| Rust | `tracing` with spans and structured fields |

The human owner decides which mechanism applies when no logging mechanism is already present. Record that decision here before code tasks proceed.

---

## Task Logging Checklist

- [ ] Code task identifies affected execution paths and boundaries
- [ ] Logs include correlation or trace context where the request/workflow crosses boundaries
- [ ] Log levels are chosen by criticality: TRACE, DEBUG, INFO, WARN, ERROR, FATAL
- [ ] External calls and dependency interactions are observable without exposing secrets
- [ ] Errors and retries are logged with safe context
- [ ] Tests or review evidence verify logging behavior where practical
